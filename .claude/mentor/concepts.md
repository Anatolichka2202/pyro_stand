# Ключевые концепции проекта — объяснения

## 1. Аппаратный таймер из байтового потока

**Ситуация:** нужно знать точное время каждого события с точностью до 1 мс.
Системные часы ОС имеют погрешность 15-16 мс (Windows) из-за планировщика.

**Решение:** датчик генерирует ровно 1 байт каждую 1 мс — аппаратно, без участия ОС.
Счётчик `absoluteIndex` не зависит от системного времени — это байтовый курсор потока.

```
absoluteIndex = 0    → порт только что открылся
absoluteIndex = 5000 → прошло ровно 5 секунд (аппаратных)
absoluteIndex = N    → N миллисекунд от точки drain
```

**Почему это лучше `QDateTime::currentMSecsSinceEpoch()`:**
- Нет дрейфа от планировщика ОС
- Нет прерываний, меняющих точность
- Точность определяется кварцем датчика, а не Windows

## 2. Drain = обнуление часов, а не очистка мусора

Пока оператор готовил стенд, буфер COM-порта накапливал байты.
1000 байт = 1000 мс = 1 секунда "прошлого" в absoluteIndex.

```
Без drain:     absoluteIndex=0 ↔ 1 секунда назад
После drain:   absoluteIndex=0 ↔ "сейчас"
```

Drain в `readingThread` выполняется **после** `sendToBoard()` — то есть после отправки
циклограммы на БЦВМ. Так `absoluteIndex=0` максимально близко к "борт получил задание".

---

## 3. Самовычитание погрешности drain

Drain не идеален — между отправкой и открытием порта проходит несколько миллисекунд.
Назовём эту погрешность `ε`.

```
syncIndex  = реальный T0 + ε
firedTick  = реальное время события + ε

deviation = firedTick - syncIndex
          = (реальное время события + ε) - (реальный T0 + ε)
          = реальное время события - реальный T0
```

`ε` исчезает. Именно поэтому отклонение считается от `syncIndex`, а не от `timeToStartMs`.

## 4. std::atomic<bool> vs bool в потоках

В `stand.h`:
```cpp
std::atomic<bool> m_running{false};
std::atomic<bool> m_stopped{false};
```

**Почему не просто bool?**
Без `atomic`, два потока (main + readingThread) читают/пишут одну переменную без
гарантий. Процессор может кэшировать старое значение, компилятор может переставить
инструкции. Результат: `readingThread` никогда не видит, что `m_stopped = true`.

`atomic` гарантирует:
- visibility: изменение видно всем потокам сразу
- ordering: нет переупорядочивания вокруг атомарной операции

## 5. Qt::QueuedConnection — почему важно

Сигналы из `readingThread` (не-Qt поток) в слоты MainWindow:
```cpp
connect(stand, &Stand::eventFired, this, [this](...){...});
// Тип соединения: Qt::AutoConnection
// readingThread → не main thread → автоматически становится QueuedConnection
```

**QueuedConnection** помещает вызов в очередь событий главного потока.
Слот выполняется в главном потоке, когда Qt обработает событие из очереди.

Без этого — прямой вызов из worker-потока обращался бы к виджетам из чужого потока.
Qt виджеты не thread-safe. Программа падала бы.

## 6. Dependency Injection через интерфейс ISerialPort

```cpp
class ISerialPort {
    virtual bool open(const QString &portName, int baudRate) = 0;
    virtual QByteArray read(qint64 maxSize) = 0;
    // ...
};

class RealSerialPort : public ISerialPort { /* QSerialPort внутри */ };
class MockSerial    : public ISerialPort { /* скриптованные события */ };
```

Stand принимает `unique_ptr<ISerialPort>`:
```cpp
Stand(QObject*, portName, unique_ptr<ISerialPort> port, SessionLogger*);
```

**Результат:** тесты передают MockSerial и проверяют логику без железа.
Demo-режим передаёт MockSerial с `setRealtime(true)` — человекопонятная пауза.

Это называется **Dependency Injection** — зависимость (порт) не создаётся внутри,
а передаётся снаружи. Код становится тестируемым без аппаратуры.

## 7. Разделение Stand / MainWindow

```
Stand (QObject)         ←→ сигналы/слоты ←→     MainWindow (QMainWindow)
 - вся бизнес-логика                              - только UI
 - readingThread                                  - connectStand() зашивает всё
 - analyzeEvents()                                - m_displayEvents
 - нет знания о виджетах                          - нет знания о COM/UDP/TFTP
```

**Почему важно:** Stand можно запустить без UI (pyro_service — JSON в stdout).
MainWindow можно тестировать с подставным Stand (tst_gui).
Слои полностью независимы.

## 8. Три закона UI-дизайна (80% результата)

### Закон 1: Контрастность важнее цвета
Текст на фоне должен иметь ratio ≥ 4.5:1 (WCAG AA).
Серый #8b949e на тёмном #161b22 → ratio ≈ 4.6. Именно так выбраны цвета.

### Закон 2: Семантика цвета (светофор)
- Зелёный = хорошо / норма
- Жёлтый = предупреждение / допуск
- Красный = ошибка / критично

Нарушать нельзя — пользователь считывает цвет рефлекторно, до чтения текста.

### Закон 3: Плотность информации
Показывай данные только тогда, когда они значимы. "—" лучше чем "—0 мс" или пустая строка.
Пустое место тоже несёт смысл: "здесь ещё ничего не произошло".
