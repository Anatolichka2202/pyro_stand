# Технические крайние случаи — pyro_stand

Документ описывает все нештатные и пограничные сценарии работы системы: текущее поведение кода, что видит оператор, и что система должна делать.

---

## 1. Задержка T0 (КП опаздывает)

**Сценарий**: контакт подъёма (канал 8, `0x80`) срабатывает позже планового времени `START_UTC_TIME`.

**Причина**: неточная синхронизация БЦВМ, реальное опоздание старта, задержка drain-окна.

**Текущее поведение**:
- `absoluteIndex` превышает `timeToStartMs`, фаза переходит в `Running`
- КП не пришёл в ожидаемое окно → помечается в `pendingChecks` как пропущенный (`status=fail`)
- Когда бит `0x80` приходит позже → **late-detection** перехватывает, `ev.firedTick` = реальный тик
- `m_syncIndex` = реальный тик КП (не плановый)
- В `analyzeEvents`: `status = "late"`, `deviationMs = |firedTick - timeToStartMs|`
- Все остальные события отмеряются от `m_syncIndex` → отклонения корректны
- UI: строка КП показывает **КП ОТКЛ** (если > 1000 мс), иначе **КП ШТАТНО**

**Тест**: `tst_reading::delayedT0_syncIndexIsActualTick`, `delayedT0_analysisStatusIsLate`

**Ограничение**: если КП опаздывает более чем на `FLIGHT_SAFETY_MS` (5000 мс) после последнего события циклограммы, поток успевает завершиться до прихода КП → КП не будет зафиксирован.

## 2. Сброс во время активного полёта (фаза Running)

**Сценарий**: оператор нажимает СТОП → затем СБРОС, пока рабочий тред ещё обрабатывает последние события.

**Текущее поведение**:
- `stop()`: `m_stopped = true`, отправляет `"STOP"` на БЦВМ, `m_running = false`, join()
- Рабочий тред завершается, может успеть поставить `flightComplete` в очередь событий Qt
- `resetForNewTest()`: join (no-op, тред уже закончен), сбрасывает все поля включая `channelCalcMs/channelTicks`
- `loadCyclogram()`: синхронно пересоздаёт события, обновляет `m_phase = Loaded`
- Устаревший `completeFlight()` из очереди: **проверяет `m_phase == Loaded` → ранний выход**
- Таблица остаётся в корректном состоянии

**Ключевой инвариант**: `completeFlight` выполняется через `QueuedConnection`, поэтому он не может исполниться раньше, чем `onReset()` вернётся в event loop. К этому моменту `m_phase` уже `Loaded`.

## 3. Обрыв COM-порта во время полёта

**Сценарий**: USB-кабель датчика отключается или порт зависает.

**Текущее поведение**:
- `RealSerialPort::waitForReadyRead()` возвращает `false` → `m_port->hasError()` проверяется
- Если ошибка: `emit portError(msg)` из рабочего треда (через AutoConnection → очередь)
- `MainWindow`: `portError` → `QMessageBox::critical`, индикатор COM красный, кнопки блокируются
- Рабочий тред завершается → `flightComplete` НЕ эмитируется (нет флага `started`)
- Анализ невозможен: UI остаётся с частично заполненными событиями

**Что оператор должен делать**: исправить подключение, перезапустить приложение (сброс без COM не даст запустить новый полёт).

**Тест**: `tst_reading::portDropMidFlight`

## 4. БЦВМ недоступна при загрузке циклограммы

**Сценарий**: `pingBcvm()` возвращает `false`.

**Текущее поведение**:
- `sendToBoard()` эмитирует `transferProgress("error")` и `logMessage("ОШИБКА: БЦВМ недоступна")`
- `transferProgress("error")` → UI показывает "Ошибка загрузки" (красный, 8 сек)
- Рабочий тред НЕ запускается, фаза остаётся `Loaded`
- Таймер обратного отсчёта не стартует

**Что оператор должен делать**: проверить сеть, убедиться что БЦВМ отвечает на `ping 192.168.17.246`.

---

## 5. TFTP таймаут или ошибка передачи

**Сценарий**: БЦВМ отвечает на ping, но TFTP-сервер не отвечает на WRQ или теряет блок.

**Текущее поведение**:
- `sendCyclogramTftp()` ждёт ACK 5000 мс (`TIMEOUT_MS`) на каждый блок
- Таймаут любого блока → `logMessage("TFTP: таймаут ACK блока N")`, return false
- `sendToBoard()` получает false, эмитирует `transferProgress("error")`
- Рабочий тред НЕ запускается, фаза остаётся `Loaded`
- Статус-лейбл: "Ошибка загрузки" (красный)

**Известное ограничение**: TFTP ERROR-пакеты (opcode 5) частично обрабатываются — ошибка логируется, но без отдельного пользовательского сообщения.

**Рекомендация**: при повторных сбоях TFTP переключиться на UDP.

## 6. Финальный TFTP-блок менее 512 байт

**Сценарий**: размер циклограммы не кратен 512 байтам (типичный случай).

**Текущее поведение**:
- Последний блок `chunk.size() < 512` → флаг `lastBlock = true`
- Блок отправляется, ACK получен → цикл завершается без отправки лишнего пустого блока
- Корректно реализовано согласно RFC 1350: последний блок < 512 байт = признак конца файла

---

## 7. Событие без маппинга каналов

**Сценарий**: ключ события в `cyclogram.ini` не совпадает ни с одной строкой маппинга.

**Текущее поведение**:
- `ev.hasChannels = false`, `ev.channels = "—"`
- Событие парсится и отображается в таблице (строка с прочерками в КАНАЛЫ/ФАКТ/ОТКЛ/СТАТУС)
- В `readingThread` событие не отслеживается (нет маски для проверки)
- При анализе: `status` остаётся `"pending"` или `"fail"`, но в UI-лог НЕ выводится (только в файл)
- В таблице статус после анализа: прочерк

**Назначение**: события типа `START_UTC_TIME` и немаппированные технические строки не создают красного шума в UI.

---

## 8. Несколько каналов в группе — разброс (channel spread)

**Сценарий**: несколько каналов в маске срабатывают не одновременно (напр. каналы 3, 4, 5 с разницей 1 мс).

**Текущее поведение**:
- `bitFirstSeen[b]` фиксирует первое появление каждого бита в потоке
- `firedAtFor(mask)` = max(bitFirstSeen) для бит маски = тик последнего канала группы
- `ev.channelSpreadMs` = max - min среди bitFirstSeen бит маски
- В колонке КАНАЛЫ: `3, 4, 5  (Δ1мс)` если spread > 0
- После анализа: подстроки `↳ Канал N` с индивидуальным `channelCalcMs`, последний канал = `← послед.`
- `firedTick` события = тик последнего канала (наихудший случай для отклонения)

## 9. КП не срабатывает вообще

**Сценарий**: канал 8 не приходит за всё время полёта.

**Текущее поведение**:
- По завершению треда: `!m_syncFound && started` → fallback `m_syncIndex = timeToStartMs`
- `emit flightComplete()` → `completeFlight()` → `analyzeEvents` с `syncIndex = timeToStartMs`
- Все отклонения считаются от планового T0
- В лог: "ВНИМАНИЕ: синхроимпульс (канал 8) не найден. Используется расчётное T0."
- Событие `LIFT_OFF_CONTACT` в таблице: `status = "fail"`, ФАКТ = прочерк
- Остальные события анализируются корректно (от расчётного T0)

---

## 10. Сброс до начала обратного отсчёта (фаза Loaded → Сброс)

**Сценарий**: циклограмма загружена, рабочий тред ещё не запущен, оператор решает перезагрузить.

**Текущее поведение**:
- Кнопка СБРОС доступна только в фазах `Completed` / `Stopped`
- В фазе `Loaded` кнопка ЗАГРУЗИТЬ снова вызывает `sendToBoard()`, который перезапускает тред
- Перезагрузка циклограммы в фазе Loaded: нажать ЗАГРУЗИТЬ повторно (не СБРОС)

**Инвариант**: СБРОС никогда не вызывается во время активного полёта — кнопка скрыта в фазах Countdown/Running.

---

## 11. Двойной клик ЗАГРУЗИТЬ

**Сценарий**: оператор нажимает ЗАГРУЗИТЬ дважды, пока идёт отсчёт.

**Текущее поведение**:
- `sendToBoard()` первым делом проверяет `if (m_running) { m_running=false; join(); }` — убивает предыдущий тред
- Затем запускает новый тред, drain буфера, отсчёт начинается заново
- Это безопасно, но приводит к потере данных предыдущего прогона

**Защита в UI**: кнопка ЗАГРУЗИТЬ отключается в фазах Countdown/Running (через `setPhase`).

---

## 12. Очень большая циклограмма (> 512 байт)

**Сценарий**: файл `cyclogram.ini` больше одного TFTP-блока.

**Текущее поведение**:
- `sendCyclogramTftp()` делит данные на блоки по 512 байт
- Каждый блок нумеруется (blockNum 1..N) и ожидает ACK с соответствующим номером
- Рассинхрон ACK → logMessage с ошибкой, return false
- Типичная циклограмма ~ 200–400 байт → один блок, работает всегда

## 13. Windows scheduler jitter при drain

**Сценарий**: Windows не является RTOS; между `sendToBoard()` и открытием COM-порта проходит непредсказуемое время (15–50 мс типично).

**Текущее поведение**:
- `drain` выполняется сразу при открытии порта в рабочем треде: сбрасывает накопленные байты
- `absoluteIndex = 0` после drain
- Любая погрешность drain (+N байт) одинаково смещает `m_syncIndex` и `firedTick` каждого события
- Вычитание `firedTick - m_syncIndex` самовычитает погрешность drain
- Остаток погрешности < 1 мс при нормальной работе ОС

**Не является проблемой**: дизайн системы намеренно устроен так, чтобы абсолютные тики не имели значения — только разности.

---

## 14. Переполнение absoluteIndex

**Сценарий**: непрерывная запись более ~2 147 483 647 мс (~24.8 суток).

**Текущее поведение**:
- `absoluteIndex` — `int64_t` (знаковый 64-бит): переполнение через ~292 миллиона лет
- `firedTick` в `EventRow` — `int` (32-бит): переполнение через ~24.8 дня
- Реальная сессия: минуты, не дни → переполнение невозможно в штатном режиме
